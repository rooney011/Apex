# F1 Lap Fingerprints

**A generative-art project that turns Formula 1 telemetry into unique visual signatures — one fingerprint per driver, per lap, per moment in time.**

---

## 1. Vision

Every fast lap in Formula 1 is a story told in throttle, brake, steering, and speed. Today that story exists only as raw data — millions of numbers per second, locked inside engineering dashboards. **The goal of this project is to translate that data into something visually beautiful, emotionally legible, and shareable — something a non-F1 fan would frame on a wall, and a fan would recognize as "exactly Senna's style."**

Think Spotify Wrapped, but for racing laps. Each lap becomes a unique generative artwork — a _fingerprint_ — driven entirely by the driver's inputs and the car's response.

---

## 2. Why this project

- **The data is free and rich.** FastF1 (open-source Python library) gives full telemetry for every F1 lap since 2018. Millions of data points per session.
- **The visual gap is real.** F1 visualization today is either broadcast TV (mass, shallow) or engineer dashboards (deep, insider). There is almost no bridge between them in the form of _art_.
- **The audience is right.** Drive to Survive pulled in a generation that responds to aesthetics and identity, not telemetry charts.
- **Plays to algorithmic strengths.** The output is generated by math from data — meaning aesthetic quality is mostly determined by the algorithm, not by hand-tuning each frame.

---

## 3. Success Criteria

The project succeeds if:

1. **Visual quality:** At least one fingerprint output is good enough that a non-F1 fan would describe it as "beautiful" with no F1 context.
2. **Identity:** Two different drivers' fingerprints of the same lap are _visibly different_ in a way that reflects their driving style.
3. **Shareable:** Output is a clean image (or short video) that works as a poster, social post, or web embed.
4. **Extensible:** The system can generate a fingerprint for any lap of any race since 2018 without manual tweaking.

Stretch success:

- A small web gallery showcasing a curated set of famous laps.
- A 3D version that holds up cinematically.
- An interactive "pick a lap" tool.

---

## 4. Technical Approach

**Data → Algorithm → Visual.**

- **Data layer:** FastF1 (Python) pulls telemetry for any lap: time, speed, throttle %, brake on/off, steering angle, gear, RPM, DRS, plus GPS position around the track.
- **Algorithm layer:** Map each telemetry channel onto a visual parameter (radius, color, thickness, height, frequency, etc.). The mapping itself is the creative work.
- **Visual layer (2D first, 3D later):** Render via Python (matplotlib / SVG) for 2D prototypes, then Three.js or Blender Python for 3D.

**Core design principle:** The shape is determined by _the lap itself_. No two laps produce the same fingerprint. The driver's inputs literally draw the artwork.

## 5. Tech Stack

### Phase 1-2 (2D)

- **Python 3.11+** — data + algorithm
- **FastF1** — telemetry source
- **NumPy, Pandas** — data processing
- **Matplotlib** + **SVG output** — 2D rendering
- **Optional: p5.js** — for browser-based 2D if we want interactivity early

### Phase 3+ (3D)

- **Three.js + React Three Fiber** — browser-based 3D, easiest to share
- **Or Blender + Python (`bpy`)** — for offline cinematic renders
- **GLSL shaders** — if we need custom material effects

### Phase 5 (Gallery / Sharing)

- **Vite + React** — lightweight static site
- **Vercel / Netlify / GitHub Pages** — free hosting
- **Optional: Cloudflare R2 / S3** — to serve pre-rendered hero images

### Tooling

- **Git** — version control from day one
- **Jupyter notebooks** — exploration phase only; production code in `.py`
- **uv / poetry** — Python env management

## 6. Phased Plan

Each phase has: **goal · deliverable · tasks · done-when · rough effort.**

---

### Phase 0 — Setup & Data Exploration

_Get the data flowing. No visuals yet._

- **Goal:** Pull telemetry for one famous lap, inspect every channel available, understand what we have to work with.
- **Deliverable:** A Jupyter notebook that loads a chosen lap and plots each raw telemetry channel (speed, throttle, brake, steering, gear, RPM, position).
- **Tasks:**
  1. Set up Python env, install FastF1, configure local cache directory.
  2. Pick a hero lap (suggestion: **Hamilton Silverstone 2020 pole**, or **Verstappen Brazil 2023 pole** — both iconic, clean data).
  3. Load lap, dump telemetry to DataFrame, plot every channel separately.
  4. Document what each channel actually represents and its sampling rate.
- **Done when:** I can run one cell and see eight subplots of the hero lap's telemetry, and I understand what every channel means.
- **Effort:** 1-2 hours.

---

### Phase 1 — 2D Proof of Concept

_The first fingerprint. Ugly is fine. Recognizable is the goal._

- **Goal:** Produce the first visual fingerprint — a single 2D image generated from the hero lap.
- **Deliverable:** One PNG/SVG image. Algorithmically generated. From real telemetry.
- **Tasks:**
  1. Sketch 2-3 visual concepts on paper _before coding_ (radial/circular, linear ribbon, abstract spline). Pick one.
  2. Build the data-to-visual mapping. Example for radial: angle = lap progress, radius = speed, color = throttle, line thickness = steering angle.
  3. Render. Iterate. Save output.
- **Done when:** We have an image where the shape clearly responds to the data (a slow corner makes the radius dip, a flat-out straight makes a smooth arc, etc.).
- **Effort:** 3-5 hours.
- **Risk:** First output will look bad. That's expected and fine.

---

### Phase 2 — 2D Aesthetic Refinement

_Make it actually look good. This is where taste matters._

- **Goal:** Take the proof of concept and iterate on it until it crosses the "would a non-fan call this beautiful?" threshold.
- **Deliverable:** A polished 2D fingerprint, plus 2-3 alternative styles for the same lap.
- **Tasks:**
  1. Iterate on color palettes (suggestion: 3-4 candidate palettes — monochrome ink, neon cyberpunk, vintage racing poster, minimalist Bauhaus).
  2. Tune line weights, smoothing, anti-aliasing, layout.
  3. Try alternate mappings (what if brake becomes glow? what if steering becomes a perpendicular oscillation?).
  4. Test on at least 3 different laps to confirm style holds up across data.
  5. Pick the winning style.
- **Done when:** I'd pay to print one of these. (Yes, that's the bar.)
- **Effort:** 6-10 hours. Most time spent iterating on screenshots back-and-forth.

---

### Phase 3 — 3D Extrusion

_Lift the 2D into space._

- **Goal:** Take the polished 2D fingerprint and produce a 3D version that holds up.
- **Deliverable:** A 3D rendering of the same lap, viewable in a browser (Three.js) or as a rendered image/video (Blender).
- **Tasks:**
  1. Decide platform: **Three.js (interactive, web-shareable)** or **Blender (cinematic stills)**. Recommend Three.js for first pass.
  2. Lift the 2D shape into 3D by mapping a new channel to z-axis (e.g., throttle = height, or G-force = vertical displacement).
  3. Build basic scene: camera, lighting, materials, background.
  4. Add rotation/animation if web-based.
  5. Iterate on screenshots — this is where I'm most blind, so expect frequent screenshot loops.
- **Done when:** The 3D version conveys _more_ than the 2D, not less. If it just looks like 2D with shadows, we stop and rethink.
- **Effort:** 8-15 hours. The screenshot loop slows iteration.

---

### Phase 4 — Multi-Lap / Driver Comparison

_The thing that makes fans care._

- **Goal:** Generate fingerprints for multiple drivers / laps and compare them side by side.
- **Deliverable:** A comparison view — two or more fingerprints rendered together, showing how driving style varies.
- **Tasks:**
  1. Pick 3-5 historic laps to compare (e.g., Senna 1988 Monaco vs Verstappen 2023 Monaco, or teammates same race).
  2. Render each as a fingerprint using the locked-in style.
  3. Build a layout: grid, slider overlay, or "ghost" overlay where two fingerprints share the same canvas.
  4. Add minimal labels (driver, year, track).
- **Done when:** The comparison makes the _style differences_ immediately visible, not just the lap times.
- **Effort:** 4-6 hours.

---

### Phase 5 — Web Gallery & Sharing

_Make it exist on the internet._

- **Goal:** A simple static website hosting the project — gallery, story, downloads.
- **Deliverable:** Live URL.
- **Tasks:**
  1. Scaffold a Vite + React (or plain HTML+JS) site.
  2. Curated gallery: 10-20 hero fingerprints with short captions.
  3. "About" page explaining what each fingerprint _is_ (what channels map to what).
  4. Download links for high-res prints.
  5. Deploy to Vercel / Netlify / GitHub Pages.
- **Done when:** A friend who doesn't know F1 can land on the site, scroll, and want to share it.
- **Effort:** 6-10 hours.

---

### Phase 6 (Stretch) — Interactive Generator

_Let users pick a lap and watch their fingerprint generate live._

- **Goal:** An interactive "pick any lap" tool on the site.
- **Deliverable:** A page where a user picks year → race → driver → lap, and a fingerprint generates in the browser.
- **Tasks:**
  1. Pre-process telemetry for a curated set of races into compact JSON.
  2. Build a UI for selection.
  3. Port the fingerprint rendering algorithm to JavaScript (Three.js or Canvas).
  4. Add "share" / "download" / "tweet your fingerprint" affordances.
- **Done when:** A user can generate their favorite driver's signature lap and share it.
- **Effort:** 15-25 hours.

---

### Phase 7 (Far Stretch) — Cinematic Edition

_The poster version._

- **Goal:** A small set of cinematic-quality renders for print / large-format.
- **Deliverable:** 5-10 hero pieces rendered in Blender at print resolution, with proper materials/lighting/composition.
- **Tasks:**
  1. Port the algorithm to Blender Python (`bpy`).
  2. Design lighting + materials per piece.
  3. Render at 4K+ resolution.
  4. Optionally produce short turntable videos.
- **Done when:** Frame-worthy. (Honest caveat: this phase benefits most from a human with taste doing material/lighting passes — I write the code, you'd direct the look.)
- **Effort:** 20+ hours. Optional.

## 8. Risks & Open Questions

**Known risks:**

- **Aesthetic blindness on my side.** I can't see what I render. Mitigation: screenshot-driven iteration in Phases 2-3.
- **3D may not add value over 2D.** Mitigation: explicit gate in Phase 3 — if it doesn't beat 2D, stop and improve 2D instead.
- **Algorithm picks the wrong channel mappings.** Mitigation: Phase 1 produces multiple sketches; we choose, not commit blindly.

**Open questions to answer in Phase 0-1:**

- Radial vs linear vs abstract layout — which best preserves the _feel_ of a lap?
- Which channel becomes the "spine" of the fingerprint? (Candidates: speed, throttle, GPS position.)
- Is one lap enough per fingerprint, or do we layer multiple laps to show consistency?

---

## 9. Out of Scope (For Now)

- Live race telemetry (FastF1 covers post-session only, and live data requires FIA agreements).
- Pre-2018 races (limited telemetry — possible but more friction).
- Audio/sound design (interesting but a separate project).
- Mobile app (web-first; native later if at all).
- Selling prints / commercial product (focus on artifact first, distribution later).
